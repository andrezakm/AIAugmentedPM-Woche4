Das PM-Pattern für die KI-Ära

AI-Augmented PM — Woche 4

Das Problem

LLMs machen einfach weiter.

Sie fragen nicht nach.

Sie füllen deine Lücken — ohne dir zu sagen, dass da welche sind.

Das ist kein Bug. Das ist das Design.

Was du brauchst

Klarheit — bevor du dem System Kontrolle gibst

Messbarkeit — bevor du "fertig" sagst

Ein Format — das Mensch und Maschine lesen können

Spec / Eval

Spec beschreibt was gebaut wird

Eval beschreibt wann es gut genug ist

Zwei Markdown-Dateien. Kein Framework. Kein Tool.

1. Klarheit für dich selbst

  • Wenn du es nicht als Spec schreiben kannst, verstehst du es noch nicht
  • Wenn du keine Eval-Kriterien formulieren kannst, weißt du nicht wann du fertig bist
  • Das LLM merkt das nicht — du musst es merken
Spec/Eval ist ein Denkwerkzeug, nicht nur ein Dokument.

2. Funktioniert auf jeder Ebene

EbeneSpecEval
Feature"Filterbare Cluster-Ansicht"10 pass/fail Kriterien
Produkt"Pattern-Agent für Feedback-Analyse"Nutzer findet Top-3 Themen in <2 Min
Strategie"Produktisierbare KI-Agenten"3 Kunden nutzen Agent eigenständig

Gleiche Frage auf jeder Ebene: Kann ich es beschreiben? Kann ich es messen?

3. Direktes Austauschformat

PM schreibt Spec + Eval

Dev liest dieselbe Datei. Erweitert technische Details.

  • Spec → User Story / Ticket in Linear oder Jira
  • Eval → Acceptance Criteria — direkt testbar
  • Kein Übersetzungsschritt. Kein "was meintest du damit?"

Die Spec — ein Beispiel

Feedback Cluster Viewer

ZweckCluster sichten und nach Quelle filtern
NutzerPMs, 1–2 Personen, nach Analyse-Run
Datenclusters.md — aus dem Pattern-Agent
UISidebar-Filter, Cluster-Karten, Belege-Expander
ConstraintsLokal, kein Backend, nur Lesen

Was, nicht Wie. Keine Zeile Code. Keine Implementierung.

Brief → Spec → Eval → Prototyp

Mensch prüft manuell  →  Agent prüft automatisch  →  Mensch vergleicht

Du entscheidest. Nicht das Modell.

Die Kette der Kontrolle

  • Level 1: Agent baut alles — du prüfst das Ergebnis
  • Level 2: Du prüfst manuell — Eval als Checkliste
  • Level 3: Agent prüft automatisch — eval-runner

Die Frage ist nicht "automatisch oder manuell?"

Die Frage ist: Wem vertraust du wann — und warum?

Wenn etwas nicht stimmt

Brief → Spec → Eval → Code

Fix immer so früh wie möglich in der Kette.

  • App zeigt Falsches? → Vielleicht ist die Spec falsch, nicht der Code
  • Eval sagt FAIL? → Vielleicht ist das Kriterium schlecht formuliert
  • Alles PASS aber es fühlt sich falsch an? → Der Brief war unklar

Der PM-Skill der Zukunft

Nicht: Code schreiben

Nicht: Prompts optimieren

Klar beschreiben. Messbar machen. Prüfen.

Spec/Eval ist das Pattern dafür.

Drei Dinge zum Mitnehmen

1. Wenn du es nicht spezifizieren kannst, ist es nicht klar

2. Wenn du es nicht evaluieren kannst, weißt du nicht wann du fertig bist

3. Spec + Eval = gemeinsame Sprache zwischen PM, Agent und Dev
